# Скрипт запуска проекта — от нуля до прода

> Основан на реальном опыте WhatDaDev (июнь 2026).  
> Универсален для любого B2B-сайта или лендинга.

---

## Фаза 0 — Интервью с заказчиком

Цель: понять бизнес, аудиторию, цель сайта. До этого — никакого кода.

### Блок 1: Бизнес
- Чем занимается компания? Один абзац своими словами.
- Кто клиент? Кто принимает решение о покупке?
- Что клиент ищет прямо сейчас (боль)?
- Как сейчас находят вас? (сарафан / реклама / SEO)
- Главный конкурент? Чем вы лучше?

### Блок 2: Сайт
- Зачем сайт? (лидген / доверие / SEO / продажи напрямую)
- Главное целевое действие: заявка / звонок / демо / покупка?
- Есть ли уже сайт? Что с ним не так?
- Какие сайты нравятся визуально? (3 примера)
- Что точно НЕ хотим?

### Блок 3: Контент
- Есть ли тексты, фото, видео? Или всё с нуля?
- Сколько страниц нужно на старте?
- Есть ли отзывы, кейсы, цифры?
- Логотип — есть в векторе (SVG/AI)?

### Блок 4: Технические ограничения
- Бюджет на хостинг/инфраструктуру?
- Нужна ли CMS (редактировать контент самому)?
- Есть ли домен? Почта на домене?
- Нужна ли интеграция с CRM / формами?

**Выход:** документ `docs/00b-discovery.md` с ответами + список того, чего не хватает.

## Фаза 1 — Брейншторм структуры

Цель: утвердить структуру сайта до дизайна и кода.

### Шаг 1: Воронки
Определить сколько воронок и какие:
```
Услуги → [что делаем] → заявка
Продукты → [что продаём] → демо/покупка
```

### Шаг 2: Карта страниц (sitemap)
Набросать на бумаге или в чате:
```
/ — главная
/services/ — услуги
  /services/vnedrenie/ — конкретная услуга
/products/ — продукты
  /products/express/ — конкретный продукт
/industries/ — отрасли
/press/ — пресс-центр / блог
/company/about/ — о команде
/contacts/ — контакты
```

### Шаг 3: Приоритеты
- Какие 3 страницы самые важные для бизнеса?
- Какие страницы нужны сразу vs. можно добавить потом?

**Выход:** утверждённый `docs/02-sitemap-ia.md`.

## Фаза 2 — Дизайн-решения (ADR)

Цель: зафиксировать ключевые визуальные и архитектурные решения **до** кода.

### Обязательные решения:
| Решение | Варианты | Зафиксировать |
|---|---|---|
| Цветовая схема | тёмная / светлая / тёмный hero + светлый контент | ADR-0002 |
| Акцентный цвет | выбрать один | ADR-0011 |
| Шрифт | Inter / Manrope / системный | ADR-0011 |
| Стек | Astro / Next / Hugo / WordPress | ADR-0001 |
| E-commerce | да / нет | ADR-0008 |
| CMS | сразу / отложить | ADR-0006 |
| Хостинг | Cloudflare / Vercel / VPS | ADR-0003 |

Каждый ADR — 1 файл в `docs/08-decisions/`, формат:
```markdown
# ADR-XXXX: Название
**Статус:** принято
**Решение:** ...
**Причина:** ...
**Альтернативы:** ...
```

## Фаза 3 — Визуализация (до кода)

Цель: согласовать визуал без написания HTML/CSS.

### Что согласовать:
1. **Главная страница** — hero-блок: текст + CTA + изображение/видео
2. **Типографика** — размеры H1/H2/H3, шрифт, межстрочный интервал
3. **Цвета** — фон, текст, акцент, карточки
4. **Компоненты** — как выглядит карточка, кнопка, форма

### Инструменты для согласования:
- **Скриншоты референсов** — сайты конкурентов или референсы стиля
- **Figma / Excalidraw** — быстрый вайрфрейм (не обязателен)
- **Словесное описание** — «как у Linear, но светлее, акцент indigo»
- **Токены** — записать в `docs/04-design/tokens.json`:

```json
{
  "color": {
    "accent": "#4F46E5",
    "dark-bg": "#0f172a",
    "text": "#1e293b"
  },
  "font": {
    "base": "Inter",
    "scale": "clamp(1rem, 2vw, 1.125rem)"
  },
  "radius": { "card": "12px", "btn": "8px" },
  "shadow": { "card": "0 4px 16px rgba(0,0,0,.08)" }
}
```

**Выход:** токены утверждены, референсы выбраны → можно писать `global.css`.

## Фаза 4 — Первый коммит

Цель: рабочая заготовка за 1 день.

### Порядок:
1. `npm create astro@latest` → выбрать minimal
2. Настроить `wrangler.jsonc` для Cloudflare
3. Написать `src/styles/global.css` — только токены и базовая типографика
4. Создать `src/layouts/Base.astro` — хедер + футер + `<slot />`
5. Создать главную страницу `/` — только hero, без контента
6. Первый деплой: `git push origin main`

**Правило:** деплой в первый день. Потом итерации.

## Фаза 5 — Итерации

Каждая итерация по схеме:

```
Проблема / задача
  ↓
Обсуждение (брейншторм в чате)
  ↓
Решение (ADR если архитектурное)
  ↓
Реализация (Edit файлов)
  ↓
Проверка (preview_screenshot)
  ↓
Коммит + пуш
  ↓
Обновить CHANGELOG + HANDOFF
```

### Приоритеты итераций:
1. **P0** — страницы основных воронок (услуги, продукт, контакты)
2. **P1** — SEO (H1, мета, JSON-LD, sitemap)
3. **P2** — доверие (команда, кейсы, отзывы)
4. **P3** — контент (статьи, глоссарий, сравнения)
5. **P4** — UI-полировка (анимации, детали)

## Правила, которые сэкономят время

1. **Токены до компонентов** — не писать ни строчки CSS пока нет `global.css` с переменными
2. **Деплой в день 1** — потом только итерации, не «сначала доделаю потом задеплою»
3. **ADR на спорные решения** — 5 минут записать почему так, сэкономят часы споров потом
4. **CHANGELOG после каждого изменения** — иначе через неделю не вспомнишь что было
5. **Мобайл с самого начала** — не «потом адаптирую», а сразу `clamp()` и медиа-запросы
6. **Нет эмодзи в B2B** — серьёзный тон, SVG-иконки вместо 📄📋
7. **Один акцентный цвет** — не два, не три. Один.
8. **Воздух = премиум** — большие отступы, не экономить на `padding`
