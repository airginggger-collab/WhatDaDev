# LESSONS — журнал грабель (что ломалось и почему)

> **Правило (пользователь, 2026-07-10, правило №7 в `CLAUDE.md`):** когда что-то ломается — разобраться **что** именно сломалось и **почему**, затем **обязательно** записать сюда, чтобы не повторять. После записи — авто-цикл: CHANGELOG + HANDOFF → commit → push.

**Формат записи:**
- **Что сломалось:** симптом/ошибка (по возможности дословный текст).
- **Почему:** корневая причина.
- **Как не повторить:** правило/проверка на будущее.
- Дата.

Самые свежие — сверху.

## Правка файла python-скриптом по индексам: вставка сдвинула текст → дубли и обрезанный тег

- **Что сломалось:** при выносе FAQ `/coordo/` в массив скрипт разрезал `coordo.astro` по символьным индексам. В сборке оказалось **12 видимых вопросов вместо 7** (пять продублировались), а на строке 497 тег обрезался прямо посреди атрибута: `<p style="font-size:14px;col      <div class="faq reveal"…`. Сборка при этом была **зелёной**: Astro проглотил битую разметку, дефект нашёлся только подсчётом вопросов в `dist`.
- **Почему:** классическая инвалидация индексов. Скрипт (1) вычислил `start`/`end` на исходном тексте, (2) сделал `src.replace(...)`, вставив массив в фронтматтер, чем сдвинул все последующие позиции на длину вставки, (3) применил срез `src[:start] + new + src[end:]` по **старым** индексам. Срез уехал, часть исходного FAQ уцелела ниже, а начало блока откусило хвост предыдущего тега.
- **Как не повторить:** (1) Для правки исходников использовать Edit с точным совпадением строки, а не срезы по индексам: Edit падает при неоднозначности, а срез молча портит файл. (2) Если скрипт всё же нужен, никогда не смешивать `replace` и слайсинг в одном проходе; считать индексы заново после каждой мутации либо собирать новый текст с одного прохода. (3) Зелёная сборка не доказывает целостность разметки: после структурных правок считать сущности в `dist` (сколько вопросов, карточек, ссылок) и сверять с ожидаемым числом. (4) Чинить такое откатом `git checkout <file>` и переделкой, а не латанием повреждённого файла.
- 2026-07-17.

## JSON.stringify в `set:html` для JSON-LD = XSS через разрыв `</script>`

- **Что сломалось:** JSON-LD вставлялся как `<script type="application/ld+json" set:html={JSON.stringify(b)}>`. `JSON.stringify` НЕ экранирует `<`, `>`, `&`, поэтому любое CMS-редактируемое текстовое поле (SEO-description, h1 отрасли, вопрос FAQ) со значением `…</script><script>alert(1)</script>` разрывало тег и исполняло произвольный JS на публичной странице. То же — в `set:html={d.lead}`, где поле по факту простой текст.
- **Почему:** `set:html` в Astro выводит строку БЕЗ экранирования (это его назначение). `JSON.stringify` безопасен для JS-парсера, но не для HTML-парсера, который видит `</script>` внутри строки как конец тега. `set:html` для поля, которое на самом деле plain text, — всегда лишний sink.
- **Как не повторить:** (1) JSON-LD выводить только через хелпер `src/lib/jsonld.ts` (`jsonLd()` экранирует `<`/`>`/`&` в `\uXXXX`), никогда голым `JSON.stringify`. (2) `set:html` использовать ТОЛЬКО для полей, которые в CMS-конфиге объявлены как HTML (`body`/markdown); для `widget: string`/`text` — обычный `{value}`, Astro экранирует сам.
- 2026-07-14.

## Диагноз по устаревшему снимку git → ложная запись в LESSONS

- **Что сломалось:** агент решил, что коммит `0f664b6` (v0.84–0.85) «ушёл без кода», и записал сюда ложный разбор. На самом деле `0f664b6` содержал все 60 файлов: его создала **параллельная сессия** за минуты до проверки, а агент опирался на снимок `git status`, сделанный при старте своей сессии.
- **Почему:** git-состояние читалось из закэшированного снимка начала сессии, при том что в репозитории одновременно работала вторая сессия. Вывод «код не закоммичен» не перепроверили свежим `git show --stat` перед записью урока.
- **Как не повторить:** перед любым выводом о состоянии репозитория (и тем более перед записью в LESSONS/коммитом «фикса») — свежий `git log --oneline -3` + `git show --stat <commit>`, а не снимок из начала сессии. Помнить: в репо может параллельно работать вторая сессия.
- 2026-07-14.

## Взял номер версии по памяти → дубль `v0.79` в CHANGELOG

- **Что сломалось:** при stop-slop-прогоне вписал новую запись в CHANGELOG как «v0.79», а HANDOFF откатил на «v0.78», хотя репозиторий уже был на **v0.81** (дек v0.79–0.81 добавлен в промежуточных сессиях). Получилась вторая `## v0.79` — коллизия версий.
- **Почему:** взял номер версии из ранней части этой (длинной, возобновлённой) сессии по памяти, не перечитав актуальные `docs/HANDOFF.md` и `docs/CHANGELOG.md` перед правкой. Состояние репо ушло вперёд, а я этого не проверил.
- **Как не повторить:** перед любой версионной правкой — **сначала прочитать первую строку `HANDOFF.md` и первую запись `CHANGELOG.md`**, брать следующий номер оттуда, а не из памяти/контекста. Особенно в возобновлённых сессиях, где репо мог продвинуться.
- 2026-07-10.

## Deep-research workflow «сжёг» веб-лимит и вернул пусто

- **Что сломалось:** полный `deep-research` workflow (29 агентов, много параллельных WebFetch) упал — все фетчи вернули `You've hit your session limit · resets 2:30pm`, итог: 0 источников, 0 фактов, ~238k токенов впустую.
- **Почему:** харнесс делает десятки веб-запросов залпом и выбивает сессионный лимит веб-инструментов; после лимита все фетч-агенты падают, а не деградируют.
- **Как не повторить:** для ресёрча — сначала одним `WebSearch` проверить, что лимит не активен; при живом лимите не гонять тяжёлый многоагентный харнесс. Надёжнее — лёгкий ручной проход (3–5 таргетных WebFetch/WebSearch под контролем) и синтез самому; тяжёлый workflow — только на свежих лимитах.
- 2026-07-10.

## Кириллица в PPTX ломается на нестандартном шрифте (□□□)

- **Что сломалось:** дек-презентация в `.pptx` со шрифтом IBM Plex Sans/Mono показывала квадраты `□□□` вместо кириллицы на машине без установленного шрифта (часть текста — норм, часть — тофу).
- **Почему:** PowerPoint подставляет отсутствующий шрифт, а подстановка не всегда содержит кириллические глифы; встроить шрифты в pptx через pptxgenjs нельзя, а проверить рендер локально нечем (нет LibreOffice).
- **Как не повторить:** для `.pptx` использовать шрифты, гарантированно установленные с кириллицей (**Arial**, **Courier New**), а фирменные шрифты держать только в PDF/HTML (там встроены в base64). Точный вид ≠ переносимость: для редактируемого файла переносимость важнее.
- 2026-07-10.

## Cloudflare-адаптер Astro ломает статический деплой

- **Что сломалось:** после `astro add cloudflare` сборка/деплой падают с ошибкой про `public/.assetsignore`.
- **Почему:** сайт статический (SSG, Workers Static Assets через `wrangler.jsonc`, `assets.directory = ./dist`). SSR-адаптер Cloudflare навязывает воркер-рантайм, несовместимый с этой схемой.
- **Как не повторить:** **не** добавлять адаптер (`astro add cloudflare`). Деплой — только статика: `npm run build` → `dist` → автопуш в `main`. Зафиксировано в `CLAUDE.md` (раздел «Git и деплой»).
- 2026-06 (перенесено в журнал 2026-07-10).

## Бинарники в корне репозитория раздувают `.git`

- **Что сломалось:** ~24MB старых бинарников (png/jpg/pdf/mov) попали в git-историю; остаются в ней даже после удаления из трекинга (2026-06-20).
- **Почему:** курируемые/черновые ассеты клали в корень репо вместо выделенных папок.
- **Как не повторить:** никаких бинарников в корне. Курируемое → `references/` или `deliverables/`; локальный мусор/скрины → `_local-assets/` или `скрины/` (gitignored). `.gitignore` блокирует `/*.png /*.jpg /*.jpeg /*.pdf /*.mov /*.svg` в корне. Полная чистка истории (`git filter-repo` + force-push) — осознанная отдельная задача.
- 2026-06-20 (перенесено в журнал 2026-07-10).
